## Section 1: V10.2 Implementation Audit

### هل حقّق «المخزون الأسبوعي المنفصل» هدفه؟

**نعم، جزئياً.** الهدف كان: مخزون يدوي أسبوعي، منفصل تماماً عن المبيعات. هذا تحقّق:

1. **الإعداد:** `inventory_enabled` toggle في Settings → يُظهر/يُخفي تبويب المخزون في BottomNav. **يعمل بشكل صحيح** عبر SettingsContext.
2. **التذكير الأسبوعي:** `shouldShowInventoryReminder()` يفحص `last_inventory_update` كل ٧ أيام. بطاقة تذكير تظهر في HomePage. **يعمل.**
3. **التسوية:** BottomSheet يعرض كل صنف + الكمية النظرية + حقل إدخال فعلي. الفرق يُحسب ويُسجَّل كمصروف هدر. **يعمل منطقياً.**

### مشاكل إدارة الحالة

1. **`reconciliationItems` يُحمَّل مرة واحدة عند فتح الورقة:** إذا أضاف المستخدم صنفاً جديداً أثناء الورقة مفتوحة (غير محتمل لكن ممكن)، لن يظهر. **منخفض الخطورة** — الورقة تُفتح وتُغلق عادةً في ثوانٍ.

2. **`saveReconciliation` يُسجّل الفرق كمصروف بقيمة `totalGap` رقمياً:** لكن `totalGap` هو `sum(abs(gap))` — يجمع القيم المطلقة. هذا يعني أن زيادة ٣ + نقص ٥ = ٨ دينار هدر. لكن الزيادة ليست هدراً. **Bug منطقي:** يجب أن يُسجّل فقط النقص (`sum of negative gaps`)، أو يُفرّق بين «هدر» (نقص) و«فائض» (زيادة).

3. **`last_inventory_update` يُحدَّث عند الحفظ بغض النظر عن النتيجة:** حتى لو ألغى المستخدم دون تغيير، يُعتبر «تم التحديث». هذا يعني أن التذكير يختفي لـ٧ أيام حتى لو لم يعدّ شيئاً. **مقبول** — تجنّب الإزعاج المتكرر.

### اختناقات الأداء

لا اختناقات جديدة. الـreconciliation يحمّل `db.getItems()` (عادةً <٥٠ صنف) — لا مشكلة.

## Section 2: Architectural Conflicts with Layer 2 (The Manager)

### هل يتعارض مخطط `items` الحالي مع المنطق الآلي/التنبؤي القادم؟

**نعم، تعارض جوهري.** جدول `items` الحالي (v8) مصمَّم للاستخدام اليدوي:

```
items: { id, name, unit, tracking_mode, current_stock, purchase_history[], reorder_day, createdAt }
```

Layer 2 سيحتاج:

| الحاجة | الحالي | المطلوب لـLayer 2 |
|---|---|---|
| تتبّع تلقائي عند البيع | `deductOnSale()` موجودة لكن **غير موصولة** | ربط `quick_products.linked_item_id` بـ`items.id` |
| BOM (وصفة) | غير موجود | جدول `bom_components` (item_id, material_id, qty_per_unit) |
| تنبؤ بناءً على الإيقاع | `reorder_day` محسوب من `purchase_history` | يعمل لكن يحتاج ≥٢ مشتريات |
| هدر تراكمي | يُسجّل كمعاملة منفصلة عند التسوية | حقل `wastage_total` في `items` للتتبّع التراكمي |
| فصل المواد الخام عن المنتجات | `tracking_mode` فقط | حقل `item_type` ('raw'|'finished') |

### هل طبيعة «المنفصل» في Layer 1 ستسبب احتكاكاً عند «الربط» في Layer 2؟

**نعم، احتكاك محتوم:**

1. **المستخدم معتاد على عدم خصم المخزون عند البيع.** Layer 2 سيخصم تلقائياً. هذا تغيير سلوكي يحتاج تواصل واضح. **التوصية:** عند تفعيل Layer 2، عرض modal تأكيد: «من الآن، بيع كوب شاي سيخصم تلقائياً: ١ كوب + ٢غ شاي + ٥مل حليب».

2. **التسوية الأسبوعية الحالية تفترض عدم الخصم التلقائي.** الكمية «النظرية» = آخر عدّ فعلي. لكن مع Layer 2، النظرية = آخر عدّ − مبيعات. التسوية ستحسب فرقاً مختلفاً تماماً. **التوصية:** إضافة حقل `auto_deducted_qty` يحفظ ما خصمه النظام تلقائياً منذ آخر تسوية.

3. **`purchase_history` array يكبر بلا حدود.** Layer 2 سيضيف المزيد من المشتريات. **التوصية:** نقل لجدول منفصل `purchase_records` قبل بناء Layer 2.

### migrations المطلوبة لـLayer 2

```js
// v9 schema:
items: '++id, name, tracking_mode, item_type, current_stock, auto_deducted_qty, wastage_total, last_updated, createdAt'
quick_products: '++id, name, price, linked_item_id, createdAt'  // +linked_item_id
bom_components: '++id, item_id, material_id, qty_per_unit'     // new table
purchase_records: '++id, item_id, qty, unit_cost, date'         // new table (migrated from purchase_history)
```

## Section 3: Architectural Conflicts with Layer 3 (The Investor)

### هل بيانات «النظري vs الفعلي» الأسبوعية كافية لتقييم مخزون للمستثمر؟

**لا، غير كافية للأسباب التالية:**

1. **لا توجد تكلفة موحدة للصنف:** `purchase_history` يحتوي على `unit_cost` لكل عملية شراء، لكن الأسعار تتغيّر. أي تكلفة نستخدم للتقييم؟ آخر شراء؟ متوسط مرجّح؟ **التوصية:** إضافة `avg_unit_cost` محسوب تلقائياً = weighted average of last N purchases.

2. **لا توجد تصنيف للأصناف:** لا يمكن للمستثمر التمييز بين «مواد قابلة للتلف» (طعام) و«مواد غير قابلة» (علب، أدوات). التالف يحتاج خصم قيمة دوري. **التوصية:** حقل `perishable` boolean + `shelf_life_days`.

3. **الفرق الأسبوعي يُسجّل كمصروف واحد:** `saveReconciliation()` يضيف معاملة واحدة بقيمة `totalGap`. المستثمر يرى «هدر ٥٠ دينار» لكن لا يعرف أي صنف سبّب الهدر. **التوصية:** جدول `wastage_records` (item_id, qty, value, date) لتتبّع هدر كل صنف على حدة.

4. **لا يوجد تقييم المخزون في الوقت الفعلي:** `computeBalanceSheet()` يحسب `inventoryValue` من `current_stock × last_unit_cost` فقط للأصناف `exact`. لكن الأصناف `predictive` لا تُقيَّم إطلاقاً. **التوصية:** تقييم كل الأصناف بـ `current_stock × avg_unit_cost`.

### ما الذي ينقص بالتحديد؟

| المؤشر المطلوب | متوفر؟ | المصدر المطلوب |
|---|---|---|
| قيمة المخزون الحالية | جزئي (exact فقط) | `avg_unit_cost` لكل صنف |
| معدل دوران المخزون | في KPIs لكن يعتمد على COGS الكلي | يحتاج COGS لكل صنف |
| نسبة الهدر | غير متاح | `wastage_records` + `purchase_records` |
| عمر المخزون (أيام منذ آخر شراء) | متاح في `purchase_history` | يعمل |
| Dead stock (مخزون راكد >٩٠ يوم) | غير متاح | فحص `last_updated` + `current_stock > 0` |

## Section 4: Opportunities & Recommendations

### فرص لمحرّك التشخيص (SWOT)

بيانات التسوية الأسبوعية تفتح فرصاً تشخيصية قيّمة:

1. **قاعدة SWOT جديدة — «هدر مرتفع»:** إذا كان `totalGap` لآخر ٤ أسابيع > ٥٪ من قيمة المشتريات، أضف بطاقة تهديد: «هدر المخزون ٧٪ من المشتريات — راجع التخزين والصلاحية».

2. **قاعدة SWOT — «مخزون راكد»:** إذا كان صنف `current_stock > 0` و`last_updated > ٩٠ يوم`، أضف بطاقة فرصة: «لديك مخزون راكد بقيمة X — خفّض الكمية أو تصرّف فيه».

3. **قاعدة SWOT — «إيقاع شراء غير منتظم»:** إذا كان `reorder_day` محسوباً والانحراف المعياري للفترات بين المشتريات عالٍ، أضف: «مشتريات [الصنف] غير منتظمة — اعتمد جدولاً ثابتاً».

### مصائد تقنية MUST AVOID عند بناء Layer 2 و 3

1. **🔴 لا تربط `deductOnSale` بـ`quickSale` قبل إضافة `linked_item_id`:** ربط عشوائي سيخصم مخزون خاطئ. يجب أولاً: إضافة الحقل، ثم بناء UI لربط منتج POS بصنف مخزون، ثم تفعيل الخصم.

2. **🔴 لا تدمج `purchase_history` في `items` عند بناء Layer 2:** Array في صف Dexie واحد يصبح بطيئاً بعد ٥٠٠+ عملية. انقل لجدول `purchase_records` من البداية.

3. **🟡 لا تستخدم `tracking_mode` كـboolean لتحديد «هل يُخصم تلقائياً؟»:** `tracking_mode = 'exact'` لا يعني بالضرورة «خصم تلقائي». قد يعني فقط «تتبّع بالعدد». أضف `auto_deduct: boolean` منفصل.

4. **🟡 لا تعتمد على `last_inventory_update` لتحديد موعد التسوية في Layer 2:** مع الخصم التلقائي، التسوية تحتاج أن تكون متكررة أكثر (يومياً أو عند كل جلسة) وليس أسبوعياً.

5. **🟢 لا تنسَ `prefers-reduced-motion`** عند إضافة انتقالات للوضع الرابع (إذا أُضيف). الـmedia query الحالي يقلّص كل شيء لـ0.01ms وهو قاسٍ — استخدم `transition-duration: 0.01ms` فقط للحركات الزخرفية، ليس للانتقالات الوظيفية.

6. **🟢 لا تخلط `LayoutContext` بـ`TermsContext`:** ذُكر في V10.1 ويتكرر. وضع المستثمر الحالي يستخدم `mode === 'investor'` من TermsContext للتحكم في التخطيط. هذا اقتران عالٍ. **التوصية:** إنشاء `LayoutContext` منفصل قبل بناء Layer 4.
